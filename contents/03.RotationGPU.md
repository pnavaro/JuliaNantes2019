# 2D Rotation Problem: CPU vs GPU Implementation

## Introduction

This notebook demonstrates how to solve a simple 2D rotation problem using both CPU and GPU implementations. The problem serves as an excellent example of:

1. **Fourier-based methods** for efficiently solving PDEs
2. **Strang splitting** technique for operator decomposition
3. **GPU acceleration** using CUDA for computational speedup
4. **Performance comparison** between CPU and GPU implementations

The rotation problem involves advecting a Gaussian pulse as it rotates in a 2D domain. This is a standard benchmark for testing numerical schemes.

### Problem Description

We solve the 2D advection equation with a rotational velocity field:

$$\frac{\partial f}{\partial t} + \vec{v} \cdot \nabla f = 0$$

where the velocity field is a **counter-clockwise rotation**:

$$\vec{v} = \begin{pmatrix} -y \\ x \end{pmatrix}$$

This creates a rigid body rotation of the solution. The exact solution at time $t$ is obtained by rotating the initial condition:

$$f(t, x, y) = f_0(x \cos(t) + y \sin(t), -x \sin(t) + y \cos(t))$$

The solution rotates counter-clockwise while maintaining its shape perfectly (no numerical diffusion).

### Mesh Structure

```julia
"""
    2D Cartesian mesh with Fourier wavenumber arrays
    
    This structure stores:
    - Grid dimensions (nx, ny)
    - Physical coordinates (x, y arrays)
    - Fourier wavenumbers (kx, ky arrays)
    
    Periodic boundary conditions are enforced by excluding the endpoint
    of the coordinate ranges.

        Constructor for 2D periodic mesh
        
        Parameters:
        - xmin, xmax: x-domain boundaries
        - nx: number of grid points in x
        - ymin, ymax: y-domain boundaries  
        - ny: number of grid points in y
        
"""
struct Mesh
    nx   :: Int64              # Number of grid points in x-direction
    ny   :: Int64              # Number of grid points in y-direction
    x    :: Vector{Float64}    # x-coordinate grid points
    y    :: Vector{Float64}    # y-coordinate grid points
    kx   :: Vector{Float64}    # Fourier wavenumbers in x-direction
    ky   :: Vector{Float64}    # Fourier wavenumbers in y-direction
    
    function Mesh(xmin, xmax, nx, ymin, ymax, ny)
        
        # Create uniform grid excluding the right endpoint (periodic BC)
        x = LinRange(xmin, xmax, nx+1)[1:end-1]
        y = LinRange(ymin, ymax, ny+1)[1:end-1]
        
        # Create Fourier wavenumber arrays
        # Standard FFT ordering: [0, 1, 2, ..., -n/2, ..., -2, -1]
        kx  = 2π ./ (xmax - xmin) .* fftfreq(nx, nx)
        ky  = 2π ./ (ymax - ymin) .* fftfreq(ny, ny)
        
        new(nx, ny, x, y, kx, ky)
    end
end
```

- We exclude the right endpoint to enforce periodic boundary conditions
- Fourier wavenumbers are pre-computed and stored for efficient access
- The wavenumber ordering follows the convention used by FFTW

### Exact Solution Function

```julia
"""
Compute the exact solution by applying the rotation transformation

For the rotation problem, the exact solution is obtained by
rotating the initial condition by angle θ = time:

f(t, x, y) = f₀(x' , y')

where (x', y') are the rotated coordinates:
x' = x cos(t) - y sin(t)
y' = x sin(t) + y cos(t)

We use a Gaussian pulse as the initial condition:
f₀(x, y) = exp(-(x-shift)²/0.1) × exp(-(y-shift)²/0.1)

Parameters:
- time: current time t
- mesh: the 2D mesh
- shift: center location of the Gaussian (default: 1.0)

Returns:
- f: 2D array with the exact solution
"""
function exact(time, mesh::Mesh; shift=1.0)
    
    f = zeros(Float64, (mesh.nx, mesh.ny))
    
    # Iterate over all grid points
    for (i, x) in enumerate(mesh.x), (j, y) in enumerate(mesh.y)
        
        # Apply rotation matrix (clockwise rotation by -time to get the
        # coordinates where the initial Gaussian was centered)
        xn = cos(time) * x - sin(time) * y
        yn = sin(time) * x + cos(time) * y
        
        # Evaluate rotated Gaussian
        f[i, j] = exp(-(xn - shift) * (xn - shift) / 0.1) * 
                  exp(-(yn - shift) * (yn - shift) / 0.1)
    end

    f
end
```

- The Gaussian pulse is centered at $(1, 1)$ in the domain $[-\pi, \pi]^2$
- As time evolves, this pulse rotates around the origin
- At time $t$, the pulse has rotated by angle $t$ radians counter-clockwise
- We can verify our numerical solution by comparing it to this exact rotating Gaussian

### CPU Implementation with Strang Splitting

```julia
"""
Solve the 2D rotation problem on CPU using Fourier spectral method
with Strang splitting

Algorithm:
The rotation problem couples advection in x and y directions.
We use Strang splitting to decompose into simpler steps:

1. Half-step in y-direction (Fourier method)
2. Full-step in x-direction (Fourier method)
3. Half-step in y-direction (Fourier method)

For the advection equation ∂f/∂t + u·∂f/∂x = 0 with periodic BCs,
the Fourier spectral solution in one time step is:

f̂_k(t+Δt) = exp(-i k u Δt) f̂_k(t)

where f̂_k is the Fourier transform and k is the wavenumber.

Parameters:
- mesh: 2D mesh structure
- nt: number of time steps
- tf: final time

Returns:
- f: final distribution function (real-valued)
"""
function rotation_on_cpu(mesh::Mesh, nt::Int64, tf::Float64)
    
    # Calculate time step
    dt = tf / nt
    
    # Allocate distribution function (use complex for FFT compatibility)
    f = zeros(ComplexF64, (mesh.nx, mesh.ny))
    
    # Initialize with exact solution at t=0
    f .= exact(0.0, mesh)
    
    # Pre-compute exponential factors for each direction
    # These encode the spectral solution of each advection step
    
    # y-direction: u_y = -x, so phase factor is exp(i·tan(dt/2)·x·k_y)
    exky = exp.(1im * tan(dt / 2) .* mesh.x .* mesh.ky')
    
    # x-direction: u_x = y, so phase factor is exp(-i·sin(dt)·y·k_x)
    ekxy = exp.(-1im * sin(dt) .* mesh.y' .* mesh.kx)
    
    # Time stepping loop
    @showprogress for n = 1:nt
        
        # === STEP 1: Half-step in y-direction ===
        # FFT in y-direction (dimension 2)
        fft!(f, 2)
        
        # Multiply by phase factor (spectral multiplication)
        f .= exky .* f
        
        # Inverse FFT back to physical space
        ifft!(f, 2)
        
        # === STEP 2: Full-step in x-direction ===
        # FFT in x-direction (dimension 1)
        fft!(f, 1)
        
        # Multiply by phase factor
        f .= ekxy .* f
        
        # Inverse FFT back to physical space
        ifft!(f, 1)
        
        # === STEP 3: Half-step in y-direction ===
        # FFT in y-direction
        fft!(f, 2)
        
        # Multiply by phase factor
        f .= exky .* f
        
        # Inverse FFT back to physical space
        ifft!(f, 2)
        
    end
    
    # Return real part (imaginary parts are numerical errors)
    real(f)
    
end
```

**Algorithm Details:**

The Strang splitting decomposes the 2D rotation problem into 1D advection problems:

1. **First half-step (y-advection)**: The y-velocity component $u_y = -x$ advects the distribution
   
2. **Full-step (x-advection)**: The x-velocity component $u_x = y$ advects the distribution
   
3. **Second half-step (y-advection)**: Another y-advection to complete the symmetric splitting

This splitting is **second-order accurate** in time while being **spectrally accurate in space**.

### GPU Acceleration with CUDA

The same algorithm can be implemented on GPU for massive speedup. Modern GPUs can achieve 10-100× speedup for FFT-heavy operations.

```julia
using CUDAdrv
CUDAdrv.name(CuDevice(0))
```

**What this does:**
- `CUDAdrv.jl` provides access to CUDA GPU drivers
- `CuDevice(0)` refers to the first GPU device
- `.name()` prints the GPU name (e.g., "NVIDIA Tesla V100")

```julia
using CuArrays
using CuArrays.CUFFT
```

**What these packages do:**
- `CuArrays`: Provides GPU array types (`CuArray`) that work like regular Julia arrays but live on GPU memory
- `CuArrays.CUFFT`: CUDA-accelerated FFT library (much faster than CPU FFTW for large arrays)

```julia
"""
Solve the 2D rotation problem on GPU using CUDA acceleration

The algorithm is identical to the CPU version, but:
1. Arrays are allocated in GPU memory (CuArray)
2. FFT plans are created for GPU execution
3. All operations happen on the GPU
4. Only the final result is transferred back to CPU

This is MUCH faster for large arrays because:
- GPU FFTs are highly optimized
- GPU memory bandwidth is much higher
- No CPU-GPU communication during the time loop

Parameters:
- mesh: 2D mesh structure (used for dimensions and wavenumbers)
- nt: number of time steps
- tf: final time

Returns:
- f: final distribution function on CPU (real-valued)
"""
function rotation_on_gpu(mesh::Mesh, nt::Int64, tf::Float64)
    
    # Calculate time step
    dt = tf / nt
    
    # Initialize on CPU
    f = zeros(ComplexF64, (mesh.nx, mesh.ny))
    f .= exact(0.0, mesh)
    
    # === Transfer to GPU ===
    # Allocate f on GPU memory
    d_f = CuArray(f)
    
    # === Create FFT Plans ===
    # Creating FFT plans once is more efficient than planning each FFT call
    # Plans are cached and reused
    
    # Forward FFT along dimension 1 (x-direction)
    p_x = plan_fft!(d_f, [1])
    
    # Inverse FFT along dimension 1
    pinv_x = plan_ifft!(d_f, [1])
    
    # Forward FFT along dimension 2 (y-direction)
    p_y = plan_fft!(d_f, [2])
    
    # Inverse FFT along dimension 2
    pinv_y = plan_ifft!(d_f, [2])
    
    # === Transfer phase factors to GPU ===
    # Pre-compute and transfer exponential factors to GPU memory
    # This avoids transferring them from CPU every iteration
    d_exky = CuArray(exp.(1im * tan(dt / 2) .* mesh.x .* mesh.ky'))
    d_ekxy = CuArray(exp.(-1im * sin(dt) .* mesh.y' .* mesh.kx))
    
    # === Time stepping loop (all on GPU) ===
    @showprogress for n = 1:nt
        
        # === STEP 1: Half-step in y-direction ===
        # Forward FFT in y-direction (using GPU plan)
        p_y * d_f
        
        # Multiply by phase factor on GPU
        d_f .*= d_exky
        
        # Inverse FFT in y-direction
        pinv_y * d_f
        
        # === STEP 2: Full-step in x-direction ===
        # Forward FFT in x-direction
        p_x * d_f
        
        # Multiply by phase factor on GPU
        d_f .*= d_ekxy
        
        # Inverse FFT in x-direction
        pinv_x * d_f
        
        # === STEP 3: Half-step in y-direction ===
        # Forward FFT in y-direction
        p_y * d_f
        
        # Multiply by phase factor on GPU
        d_f .*= d_exky
        
        # Inverse FFT in y-direction
        pinv_y * d_f
        
    end
    
    # === Transfer result back to CPU ===
    # collect() transfers the GPU array back to CPU memory
    f .= collect(d_f)
    
    # Return real part
    real(f)
    
end
```

**GPU vs CPU Key Differences:**

| Aspect | CPU | GPU |
|--------|-----|-----|
| Array type | `Array{ComplexF64}` | `CuArray{ComplexF64}` |
| FFT library | FFTW | CUFFT |
| Memory | CPU RAM | GPU VRAM |
| FFT Planning | Done before loops | Critical for performance |
| Data Transfer | Implicit | Explicit with `CuArray` |
| Speedup | Baseline | 10-100× for large arrays |

**Optimization Principles Applied:**

1. **Pre-compute on CPU, transfer once**: Exponential factors computed once and transferred
2. **Avoid CPU-GPU transfers in loops**: All computation happens on GPU
3. **Pre-plan FFTs**: Creating plans once and reusing them is much faster
4. **Use in-place FFTs**: `fft!` modifies arrays in place, saving memory

### Performance Analysis and Interpretation

To properly compare CPU and GPU:

```julia
# CPU timing
time_cpu = @elapsed rotation_on_cpu(mesh, nt, tf)

# GPU timing (excluding first run which includes compilation)
rotation_on_gpu(mesh, 1, 0.1)  # Warmup
time_gpu = @elapsed rotation_on_gpu(mesh, nt, tf)

# Speedup factor
speedup = time_cpu / time_gpu
println("CPU time: $(time_cpu) seconds")
println("GPU time: $(time_gpu) seconds")
println("Speedup: $(speedup)×")
```

### Where GPU Wins:

- **Large arrays**: For this 1024×1024 mesh, GPU acceleration is substantial
- **Many time steps**: GPU amortizes startup costs over many iterations
- **FFT-heavy computations**: GPUs are optimized for FFT operations

### Where CPU is Competitive:

- **Small arrays**: Overhead of GPU transfer dominates
- **Few time steps**: GPU initialization not worth it
- **Memory-limited problems**: Some GPUs have limited VRAM
